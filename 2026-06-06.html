Thinking Machines Lab (He H, et al.). Defeating Nondeterminism in LLM Inference. Thinking Machines Lab — Connectionism (blog). 2025. Reference code: github.com/thinking-machines-lab/batch-invariant-ops
https://thinkingmachines.ai/blog/defeating-nondeterminism-in-llm-inference/
Overview: Engineering blog (you asked to route this one through scout). Explains why production LLM endpoints return DIFFERENT completions for the SAME prompt even at temperature 0, and ships open-source batch-invariant kernels that remove the effect. Source type: engineering blog (infrastructure, not clinical-specific). System: LLM inference servers (vLLM-class), GPU reduction kernels (RMSNorm/matmul/attention). Task type: reproducibility/determinism infrastructure — cross-cutting, applies to any evaluation. Clinical content: none (foundational LLM methodology).
Eval question: When is 'same input → same output' (numerical reproducibility) a property an evaluator actually HOLDS — versus one they ASSUME from setting temperature 0? Separates the decoding knobs from the serving-stack behavior that really controls reproducibility.
Methodology: • Diagnoses the cause: dynamic request BATCHING changes the numerical reduction order, so identical inputs round differently depending on co-tenant server load. • Shows most forward-pass kernels are bitwise-repeatable for a FIXED input shape, but inference is SYSTEM-level nondeterministic because output depends on batch size. • Implements batch-invariant RMSNorm/matmul/attention kernels (fixed reduction order, padding-safe). • Demonstrates bitwise-identical outputs across varying batch sizes / concurrency.
Measurement: Determinism criterion = bitwise-identical output across varying batch size and server load. Qualitative engineering demonstration; no clinical metric or threshold.
Key findings: • Primary cause of temp-0 nondeterminism is BATCH NON-INVARIANCE, not floating-point atomics. • With batch-invariant kernels, outputs are reproducible regardless of concurrency — at a performance cost. • Practical levers when you control the stack: temperature 0 + top_k 1 + top_p 1, FP32 where stability beats speed, and TEST UNDER REAL CONCURRENCY, not a single local loop.
Soft spots: • Non-clinical; makes no healthcare claim. • The fix requires CONTROL of the inference stack (open weights / self-host / vLLM) — NOT available on most multi-tenant commercial APIs, which is the typical clinical deployment. • Addresses only NUMERICAL reproducibility — silent on behavioral/semantic stability (paraphrase robustness, adversarial hallucination), which is the PQ-91 / PQ-59 / PQ-107 axis. • Performance overhead may be unacceptable in latency-sensitive serving.
Strengths: • Named expert; open-source, independently verifiable reference library. • Specific, falsifiable mechanism; corrects a widespread misconception ('temp 0 ⇒ deterministic'). • Engineering-honest about the speed tradeoff. • Canonical reference for a distinction the KB already half-makes (PQ-91, PQ-59).
Relevance: Layer 1 (task types): cross-cutting — every task type. Most directly reinforces Extraction §2 'Non-Determinism as Evaluation Metric' (PQ-91) and clarifies it is DISTINCT from PQ-59's behavioral (adversarial-hallucination) non-determinism. Layer 2 (kit connection): any kit that claims reproducibility BECAUSE temperature is 0 should instead MEASURE it (R repeats; report run-to-run variability). Directly relevant to VAP and CTA-PR extraction on multi-tenant endpoints, where batching variance is not user-controllable. Canonical anchor for the draft 'Two Determinisms' / 'R-Selector' one-pagers in Incoming/for planner/.
